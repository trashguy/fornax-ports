# fornax-ports

Package repository for the [Fornax](https://github.com/trashguy/Fornax) operating system, used by the `fay` package manager.

## Structure

```
fornax-ports/
├── repo.json              # package index (auto-generated by fay-build)
├── core/                  # base OS packages
│   └── init/FAYBUILD
├── extra/                 # additional packages
│   └── lua/FAYBUILD
└── posix/                 # POSIX realm C packages
    ├── less/FAYBUILD
    └── xxd/
        ├── FAYBUILD
        └── xxd.c
```

Packages are organized into categories:

- **core/** — base OS components (init, fsh, fxfs, etc.)
- **extra/** — additional native Zig packages (including hardware drivers like [fornax-amdgpu](https://github.com/trashguy/fornax-amdgpu))
- **posix/** — C programs built against the Fornax POSIX shim (musl libc)

## How packages work

Fornax has two compilation realms:

- **native** — Zig programs linked against the Fornax userspace library (`lib/fornax`)
- **posix** — C programs compiled against musl libc with a POSIX syscall shim

Packages are cross-compiled on the host by `fay-build`, producing `.tar.gz` archives. On-device, `fay install <pkg>` downloads and extracts pre-built packages. If a Zig toolchain is installed on-device, `fay install --build <pkg>` can build from source locally.

## Adding a port

Each package lives in its own directory under the appropriate category, with a `FAYBUILD` file (JSON) describing the package.

### 1. Create the directory

```
posix/mypackage/FAYBUILD
posix/mypackage/mypackage-fornax.patch   # optional
posix/mypackage/source.c                 # optional, for bundled sources
```

### 2. Write the FAYBUILD

```json
{
  "pkgname": "mypackage",
  "pkgver": "1.0.0",
  "pkgrel": 1,
  "epoch": 0,
  "pkgdesc": "Short description of what it does",
  "arch": ["x86_64"],
  "realm": "posix",
  "buildable": true,
  "depends": [],
  "makedepends": [],
  "source": ["https://example.com/mypackage-1.0.0.tar.gz"],
  "sha256sums": ["abcdef1234..."],
  "patches": [],
  "build": [
    "cd mypackage-1.0.0",
    "cc -o mypackage main.c util.c"
  ],
  "package": [
    "mkdir -p $pkgdir/bin",
    "cp mypackage-1.0.0/mypackage $pkgdir/bin/mypackage"
  ]
}
```

### FAYBUILD fields

| Field | Required | Description |
|-------|----------|-------------|
| `pkgname` | yes | Package name (lowercase, no spaces) |
| `pkgver` | yes | Upstream version (e.g. `"5.4.7"`) |
| `pkgrel` | yes | Package revision, increment for packaging changes |
| `epoch` | no | Version epoch, overrides version comparison (default `0`) |
| `pkgdesc` | yes | One-line description |
| `arch` | yes | Target architectures: `["x86_64"]`, `["riscv64"]`, or both |
| `realm` | yes | `"posix"` (C/musl) or `"native"` (Zig) |
| `buildable` | yes | `true` if it can be built from source, `false` for binary-only |
| `depends` | yes | Runtime dependencies (other package names) |
| `makedepends` | no | Build-time dependencies (e.g. `["tcc"]`) |
| `source` | no | URLs to download source tarballs |
| `sha256sums` | no | SHA-256 checksums for each source entry (same order) |
| `patches` | no | Patch files in the same directory, applied in order |
| `build` | no | Shell commands to compile (run in source directory) |
| `package` | no | Shell commands to install files into `$pkgdir` |

### 3. Source options

You can provide source code in three ways:

**Remote source** — downloaded and verified by SHA-256:
```json
"source": ["https://example.com/pkg-1.0.tar.gz"],
"sha256sums": ["abc123..."]
```

**Bundled source** — small programs can include source files directly in the port directory alongside the FAYBUILD. Leave `source` empty:
```json
"source": [],
"sha256sums": []
```

**No source** — binary-only packages (e.g. pre-built toolchains):
```json
"buildable": false,
"source": [],
"build": []
```

### 4. Build commands

The `build` array contains shell commands run by `fsh`. Available variables:

- `$srcdir` — directory containing extracted/bundled sources
- `$pkgdir` — staging directory; files here get packaged into the .tar.gz
- `cc` — C compiler (tcc or zig cc, depending on what's installed)

For posix realm packages, the C compiler links against musl libc with the Fornax POSIX shim automatically.

### 5. Patches

Place patch files in the same directory as the FAYBUILD. They're applied in the order listed:

```json
"patches": ["fix-headers.patch", "fornax-compat.patch"]
```

Patches should be in unified diff format (`diff -u` or `git diff`).

### Tips for porting

- **Start small** — single-file programs with no deps are easiest (see `posix/xxd/` for an example)
- **Avoid autoconf** — Fornax doesn't have `./configure`. Write build commands directly
- **Check syscall support** — the POSIX shim covers basics (open, read, write, close, mmap, etc.) but not everything. See `lib/posix/shim.c` in the Fornax repo for supported Linux syscall translations
- **No dynamic linking** — everything is statically linked
- **No network in builds** — build commands run offline; all sources must be downloaded beforehand
- **Test on host first** — verify your C code compiles with `zig cc` targeting freestanding before adding to ports

## Versioning

Follows Arch Linux conventions: `epoch:pkgver-pkgrel`

- Bump `pkgrel` when changing the FAYBUILD without an upstream version change
- Bump `epoch` (rare) when upstream changes version numbering schemes
- Version comparison: epoch first, then pkgver segment-by-segment, then pkgrel

## repo.json

The `repo.json` index is auto-generated by `fay-build` and contains download URLs and checksums for pre-built packages. Don't edit it by hand.
